# Replication package: a typology of unmarried cohabitation in the Czech Republic

This repository reproduces the figures and tables in the article *[full title,
authors, journal, year — to be completed by the authors]*. The analysis
describes a typology of unmarried cohabitation using Czech data from the
Generations and Gender Survey (GGS / GGP), comparing 2005 and 2021.

> **Status:** replication package prepared for deposit on the Open Science
> Framework (OSF). The micro-data are **not** distributed here (they are gated;
> see *Data availability* below). Everything else needed to reproduce the
> published exhibits from those data is included.

## 2. Data availability

The analysis uses three Czech GGS/GGP micro-data files. **These data are not
included** in this package: they are distributed by the Generations and Gender
Programme under a data-use agreement and cannot be redistributed. They are free
of charge for research after registration.

**Source:** Generations and Gender Programme — <https://www.ggp-i.org/> (data
access via the GGP Data Archive). Register, accept the data-use agreement, and
download the Czech datasets below.

**Datasets used** (versions are those used for the published results, taken from
the original download filenames):

| Pipeline file (place here)   | GGP/GGS product                                   | Version | Wave in article |
|------------------------------|---------------------------------------------------|---------|-----------------|
| `data/ggp/round1.dta`        | GGS Wave 1, Czech Republic                        | V.4.4   | "2005"          |
| `data/ggp/round2.dta`        | GGS-II (GGP-2020) Wave 1, Czech Republic          | V.2.1   | "2021"          |
| `data/ggp/pilot.dta`         | GGP-2020 Pilot, Czech Republic                    | V.0.3   | "2021"          |

After downloading and unpacking each product, save the Stata `.dta` file under
`data/ggp/` with **exactly** the filename in the first column — `analysis.qmd`
loads these three paths directly. The 2021 wave in the article pools
`round2.dta` and `pilot.dta` (the pilot has no survey weight and is assigned a
weight of 1 in the append step).

> A pilot follow-up file (`pilot_followup.dta`) may also be available from GGP.
> It is **not used** by this pipeline and is not required.

## 3. Computational requirements

- **R** version **4.6.0** (the version used for the published results).
- **R packages:** pinned in `renv.lock` (115 packages; key ones: tidyverse
  2.0.0, dplyr 1.2.1, haven 2.5.5, sjmisc 2.8.11, janitor, ggrepel 0.9.8,
  flextable 0.9.11, officer, knitr 1.51).

To recreate the exact package environment:

```r
install.packages("renv")   # if not already installed
renv::restore()            # installs the versions pinned in renv.lock
```

Pinning matters here: the `round_to_100()` largest-remainder helper breaks ties
by input row order, which depends on the row order `haven::read_dta()` returns
and on `dplyr`'s grouping. A different `haven`/`dplyr`/R version could in
principle shift a displayed integer percentage by 1pp. `renv.lock` guards
against that.

- **Fonts:** the figures use **Arial** only (no proprietary fonts). On systems
  without Arial, the figures still render but substitute a default sans-serif
  font; this affects only label typeface, not any value.

- **Operating system:** developed on Windows 11. The pipeline is OS-portable
  (relative paths throughout); no OS-specific code remains in the canonical
  scripts.

## 4. Instructions to reproduce

1. Obtain the three micro-data files and place them in `data/ggp/` (§2).
2. Restore the R environment: `renv::restore()` (§3).
3. From the package root, run the master script:

   ```sh
   Rscript run_all.R
   ```

**Why a master script?** The three `.qmd` files share a single R session.
`figures.qmd` and `tables.qmd` use objects created in
`analysis.qmd` (the `round_to_100()` helper and the data frames `analysis`,
`analysis_sample`, `figure4`, `figure5`, `figure6`); these are never written to
or read from disk. Quarto renders each `.qmd` in its own fresh R session, so the
files **cannot** be rendered independently — the downstream files would error on
missing objects. `run_all.R` extracts the R code from each file with
`knitr::purl()` and `source()`s it into one shared session, in order:

```
analysis.qmd  →  figures.qmd  →  tables.qmd
```

Running it regenerates everything in `plots/` and `tables/`.

## 6. Reproducibility notes

- **Reproducing the published exhibits.** The numbers are generated by the
  scripts; the rounding and display choices are deliberate and documented in
  `CHANGELOG.md`. A clean end-to-end run reproduces every table exactly and
  Figures 1 and 2 byte-for-byte.

- **Figures 3 and 4 use `ggrepel`.** `ggrepel::geom_text_repel()` places data
  labels with a force-directed algorithm seeded from R's RNG. With no seed set,
  label *positions* jitter by a few pixels between runs — the plotted **values
  are identical** (they match the corresponding tables), only label placement
  moves. To make these two figures byte-reproducible, set a seed before each
  plot (e.g. `set.seed(1)`); note this would change label positions away from
  the exact published PNGs, so it is left as the authors' choice.
